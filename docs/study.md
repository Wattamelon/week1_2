# 리눅스 프로세스 및 시스템 리소스 트러블슈팅 정리

# 프로세스(Process)

프로세스(Process)는 실행 중인 프로그램이다.

예를 들어:
- 크롬 실행
- 파이썬 실행
- Node.js 서버 실행

모두 프로세스로 동작한다.

운영체제는 각 프로세스에:
- PID(Process ID)
- 메모리
- CPU 시간
- 파일 접근 권한

등을 할당한다.

관련 명령어:

```bash
ps -ef
```

현재 실행 중인 프로세스 확인

```bash
ps -ef | grep python
```

특정 프로세스 검색

```bash
top
```

실시간 프로세스 상태 확인

---

# 스레드(Thread)

스레드는 프로세스 내부의 작업 실행 단위다.

프로세스:
- 하나의 실행 공간

스레드:
- 그 안에서 실제 작업 수행

예시:
- 게임 프로세스
  - 렌더링 스레드
  - 입력 처리 스레드
  - 사운드 스레드

멀티스레드:
- 여러 작업을 동시에 수행 가능
- 하지만 동기화 문제가 발생할 수 있음

관련 명령어:

```bash
top -H
```

스레드 단위 확인

```bash
ps -L
```

프로세스 내부 스레드 확인

---

# PID(Process ID)

PID는 프로세스 고유 번호다.

예시:

```bash
ps -ef
```

출력:

```bash
user   12345  ...
```

여기서:
- 12345 = PID

프로세스를 종료할 때 사용:

```bash
kill 12345
```

강제 종료:

```bash
kill -9 12345
```

---

# 시그널(Signal)

시그널은 운영체제가 프로세스에게 보내는 메시지다.

대표 시그널:

| 시그널 | 의미 |
|---|---|
| SIGTERM | 정상 종료 요청 |
| SIGKILL | 강제 종료 |
| SIGINT | Ctrl + C |
| SIGHUP | 설정 재로드 |

예시:

```bash
kill -15 PID
```

SIGTERM

```bash
kill -9 PID
```

SIGKILL

---

# 메모리 구조

프로세스 메모리는 크게:

- Code 영역
- Data 영역
- Heap 영역
- Stack 영역

으로 구성된다.

## Stack

함수 호출 시 사용된다.

특징:
- 자동 관리
- 함수 종료 시 자동 반환

## Heap

동적 메모리 영역

특징:
- 직접 해제 필요
- 해제 안 하면 메모리 누수 발생

Python도 내부적으로 Heap을 사용한다.

---

# Memory Leak

메모리 누수(Memory Leak)는:

사용하지 않는 메모리를 해제하지 않는 현상

이다.

결과:
- 메모리 사용량 계속 증가
- 시스템 느려짐
- OOM 발생 가능

미션에서:
- monitor.sh 로 메모리 증가 관찰
- MEMORY_LIMIT 도달 시 종료

---

# OOM(Out Of Memory)

시스템 메모리가 부족한 상태

결과:
- 프로세스 종료
- 시스템 불안정

Linux에서는:
- OOM Killer
- MemoryGuard

같은 보호 정책이 동작할 수 있다.

---

# CPU 사용률(CPU Usage)

CPU 사용률은:
프로세스가 CPU를 얼마나 사용하는지 나타낸다.

문제 상황:
- 특정 프로세스가 CPU 독점
- 시스템 전체 느려짐

관련 명령어:

```bash
top
```

```bash
htop
```

```bash
ps -eo pid,cmd,%cpu --sort=-%cpu
```

CPU 사용률 높은 순 정렬

---

# CPU Spike

CPU Spike:
짧은 시간 동안 CPU 사용률이 급격히 증가하는 현상

원인:
- 무한루프
- 과도한 연산
- 멀티스레드 경쟁
- 잘못된 로직

결과:
- 응답 지연
- Watchdog 종료

---

# Watchdog

Watchdog은:
시스템 보호용 감시 기능

특정 조건:
- CPU 과점유
- 무응답
- 메모리 초과

등이 발생하면 프로세스를 종료한다.

실무에서도:
- 서버 감시
- 임베디드 시스템
- 컨테이너 관리

등에서 자주 사용된다.

---

# Deadlock(교착상태)

Deadlock:
서로 자원을 기다리며 무한 대기하는 상태

특징:
- 프로세스 안 죽음
- CPU 거의 안 씀
- 로그 멈춤
- 응답 없음

---

# Deadlock 4대 조건

## Mutual Exclusion(상호 배제)

한 번에 하나만 자원 사용 가능

## Hold and Wait(점유 대기)

자원 가진 상태로 다른 자원 기다림

## No Preemption(비선점)

강제로 자원 회수 불가능

## Circular Wait(순환 대기)

서로 상대 자원 기다림

4개 모두 만족 시 Deadlock 발생 가능

---

# Dining Philosophers Problem

Deadlock 대표 예제

철학자들이:
- 왼쪽 포크
- 오른쪽 포크

를 서로 기다리며 멈추는 문제

운영체제 동기화 문제 설명에 자주 사용됨

---

# 스레드(Thread)

스레드는:
프로세스 내부에서 실제 작업을 수행하는 실행 흐름이다.

쉽게 말하면:

- 프로세스 = 하나의 프로그램 실행 공간
- 스레드 = 그 안에서 실제로 일하는 작업 단위

라고 생각하면 된다.

예시:
- 크롬 브라우저 프로세스
  - 화면 출력 스레드
  - 사용자 입력 처리 스레드
  - 다운로드 스레드
  - 동영상 재생 스레드

즉,
하나의 프로세스 안에 여러 스레드가 존재할 수 있다.

---

# 프로세스와 스레드 차이

| 항목 | 프로세스 | 스레드 |
|---|---|---|
| 의미 | 실행 중인 프로그램 | 프로세스 내부 작업 단위 |
| 메모리 | 독립적 | 공유 |
| 생성 비용 | 큼 | 작음 |
| 통신 | 느림(IPC 필요) | 빠름 |
| 안정성 | 높음 | 낮음 |

---

# 스레드의 핵심 특징

스레드는:
같은 프로세스 내부 자원을 공유한다.

공유하는 것:
- Heap 메모리
- 전역 변수
- 파일 디스크립터
- 코드 영역

공유하지 않는 것:
- Stack
- 레지스터

즉:
스레드마다 자기 Stack은 따로 가지고,
나머지 자원은 함께 사용한다.

---

# 왜 스레드를 사용하는가

## 동시에 여러 작업 처리 가능

예시:
- 웹서버
- 게임
- 채팅 프로그램
- 영상처리

하나의 프로그램 안에서:
여러 작업을 동시에 처리 가능하다.

---

# 스레드 사용 예시

## 웹서버

사용자 A 요청 처리
사용자 B 요청 처리
사용자 C 요청 처리

를 각각 다른 스레드가 수행 가능

---

## 게임

- 렌더링
- 입력 처리
- 물리 계산
- 사운드 처리

를 동시에 수행 가능

---

## 영상처리 / AI

- 영상 읽기
- 전처리
- 추론
- 저장

등을 병렬 처리 가능

---

# 멀티스레드(Multi-thread)

여러 개의 스레드를 동시에 사용하는 방식

장점:
- 빠름
- CPU 활용 증가
- 응답성 향상

단점:
- 동기화 문제 발생 가능
- 디버깅 어려움
- Deadlock 가능성 존재

---

# Stack과 Heap 관점에서의 스레드

## Stack

스레드마다 독립적

함수 호출 정보 저장:
- 지역 변수
- 함수 호출 기록
- 리턴 주소

예시:

```python
def test():
    a = 10
```

a 는 Stack에 저장된다.

---

## Heap

스레드끼리 공유

예시:

```python
data = []
```

여러 스레드가 동시에 접근 가능

그래서 충돌 위험 존재

---

# Race Condition(경쟁 상태)

여러 스레드가:
동시에 같은 데이터를 수정하려 할 때 발생하는 문제

예시:

현재 값:
10000원

스레드 A:
5000원 출금

스레드 B:
5000원 출금

동시에 처리되면:
둘 다 "10000원 있음" 이라고 판단 가능

결과:
데이터 꼬임 발생 가능

---

# 동기화(Synchronization)

스레드 충돌을 막기 위한 기술

대표 방법:
- Lock(Mutex)
- Semaphore
- Monitor

---

# Lock(Mutex)

특정 자원을:
한 번에 하나의 스레드만 사용하도록 제한

예시:

```python
lock.acquire()

# 공유 데이터 수정

lock.release()
```

의미:
- lock.acquire()
  → 자원 사용 시작
- lock.release()
  → 자원 반납

---

# Lock을 사용하는 이유

공유 데이터 보호

예시:
- 은행 계좌
- DB 데이터
- 파일 쓰기
- 캐시 수정

등에서 필수

---

# Deadlock(교착상태)

여러 스레드가:
서로 자원을 기다리며 무한 대기하는 상태

예시:

스레드 A:
- 자원 1 사용 중
- 자원 2 기다림

스레드 B:
- 자원 2 사용 중
- 자원 1 기다림

결과:
둘 다 영원히 대기

---

# Deadlock 특징

- 프로세스 안 죽음
- PID 존재
- CPU 거의 사용 안 함
- 메모리 변화 없음
- 로그 멈춤

즉:
프로그램은 살아있지만 아무 일도 못 하는 상태

---

# Deadlock 4대 조건

## Mutual Exclusion(상호 배제)

자원을 한 번에 하나만 사용 가능

## Hold and Wait(점유 대기)

자원을 가진 채 다른 자원 기다림

## No Preemption(비선점)

강제로 자원 회수 불가능

# Context Switching

# Round Robin

짧은 시간(Time Slice)씩 CPU 분배

예시:

- Thread A 실행
- Thread B 실행
- Thread C 실행
- 다시 A 실행

공평하게 번갈아 처리

# 컴퓨터구조와 운영체제 기본 이론 정리

이번 미션은 단순히 리눅스 명령어를 사용하는 과제가 아니라, 프로그램이 운영체제 위에서 어떻게 실행되고, CPU와 메모리를 어떻게 사용하며, 문제가 생겼을 때 어떤 방식으로 관찰하고 분석하는지를 배우는 과제이다.

따라서 이 미션을 제대로 이해하려면 컴퓨터구조와 운영체제의 기본 개념을 함께 알아야 한다.

---

# 컴퓨터 시스템의 큰 구조

컴퓨터는 크게 다음 요소들로 구성된다.

- CPU
- 메모리
- 저장장치
- 입출력 장치
- 운영체제
- 응용 프로그램

프로그램은 혼자 실행되는 것이 아니다.

사용자가 프로그램을 실행하면 운영체제가 해당 프로그램을 메모리에 올리고, CPU가 명령어를 하나씩 실행한다.

즉, 프로그램 실행 흐름은 대략 다음과 같다.

```text
사용자 실행 요청
↓
운영체제가 프로그램 파일을 메모리에 적재
↓
프로세스 생성
↓
CPU가 명령어 실행
↓
운영체제가 CPU, 메모리, 파일, 네트워크 자원 관리
```

이번 미션의 `agent-leak-app`도 결국 운영체제 위에서 실행되는 하나의 프로세스이다.

---

# 하드웨어와 소프트웨어의 관계

컴퓨터는 하드웨어만으로는 아무 작업도 할 수 없다.

하드웨어:
- CPU
- RAM
- SSD
- 네트워크 카드
- 키보드
- 모니터

소프트웨어:
- 운영체제
- 프로그램
- 드라이버
- 라이브러리

운영체제는 하드웨어와 프로그램 사이에서 중간 관리자 역할을 한다.

예를 들어 프로그램이 “파일을 읽고 싶다”고 하면, 프로그램이 SSD를 직접 제어하는 것이 아니라 운영체제에게 요청한다.

```text
프로그램
↓
운영체제
↓
하드웨어
```

이 구조 때문에 운영체제를 이해하면 프로그램 장애를 훨씬 잘 분석할 수 있다.

---

# CPU

CPU는 Central Processing Unit의 약자이다.

CPU는 컴퓨터의 핵심 연산 장치이다.

CPU가 하는 일은 크게 다음과 같다.

- 명령어 가져오기
- 명령어 해석하기
- 명령어 실행하기
- 결과 저장하기

CPU는 프로그램 코드를 한 줄씩 실행하는 것처럼 보이지만, 실제로는 기계어 명령어 단위로 매우 빠르게 처리한다.

---

# CPU의 기본 동작 과정

CPU는 보통 다음 과정을 반복한다.

```text
Fetch → Decode → Execute → Write Back
```

## Fetch

메모리에서 실행할 명령어를 가져온다.

## Decode

가져온 명령어가 어떤 작업인지 해석한다.

## Execute

실제 연산을 수행한다.

## Write Back

연산 결과를 레지스터나 메모리에 저장한다.

이 과정을 명령어 사이클이라고 한다.

---

# CPU 코어

CPU 코어는 실제로 명령어를 실행하는 물리적인 연산 단위이다.

예를 들어:

```text
4코어 CPU
```

라면 동시에 여러 작업을 처리할 수 있는 물리적 실행 단위가 4개 있다는 뜻이다.

하지만 코어가 4개라고 해서 프로그램 4개만 실행 가능한 것은 아니다.

운영체제가 매우 빠르게 작업을 바꿔가며 실행하기 때문에 수십, 수백 개의 프로세스가 동시에 실행되는 것처럼 보인다.

---

# CPU 사용률

CPU 사용률은 CPU가 얼마나 바쁘게 일하고 있는지를 나타내는 지표이다.

예를 들어:

```text
CPU 100%
```

는 CPU가 쉬지 않고 계속 작업 중이라는 뜻이다.

이번 미션에서 CPU Spike는 특정 프로세스가 CPU를 과도하게 사용하는 상황이다.

원인 예시:
- 무한 루프
- 너무 많은 계산
- 비효율적인 알고리즘
- 스레드 폭주
- busy waiting

---

# CPU Spike

CPU Spike는 CPU 사용률이 순간적으로 급격히 상승하는 현상이다.

예를 들어 평소에는 CPU 사용률이 5%였는데 갑자기 95%까지 올라가는 상황이다.

CPU Spike가 발생하면:
- 프로그램 응답이 느려짐
- 서버 지연 발생
- 다른 프로세스 실행에 영향
- Watchdog에 의해 종료될 수 있음

이번 미션에서는 `CPU_MAX_OCCUPY` 값을 조정하면서 CPU 과점유 상황을 관찰한다.

---

# 레지스터

레지스터는 CPU 내부에 있는 매우 빠른 임시 저장 공간이다.

CPU는 메모리보다 레지스터에 접근하는 속도가 훨씬 빠르다.

레지스터는 다음과 같은 값을 저장한다.

- 현재 계산 중인 값
- 다음 실행할 명령어 주소
- 함수 호출 상태
- 조건 비교 결과

대표적인 레지스터:
- PC 또는 IP
- SP
- BP
- General Purpose Register

---

# PC 또는 IP

PC는 Program Counter이다.

IP는 Instruction Pointer라고도 한다.

역할:
- 다음에 실행할 명령어의 주소를 저장한다.

CPU는 PC가 가리키는 주소로 가서 다음 명령어를 가져온다.

즉, PC는 프로그램 실행 흐름을 결정하는 중요한 레지스터이다.

---

# SP

SP는 Stack Pointer이다.

역할:
- 현재 스택의 맨 위 위치를 가리킨다.

함수가 호출되면 스택에 지역 변수나 반환 주소가 쌓인다.

이때 SP가 변경된다.

---

# 메모리

메모리는 실행 중인 프로그램과 데이터를 저장하는 공간이다.

보통 RAM을 의미한다.

저장장치인 SSD/HDD와 메모리는 다르다.

| 항목 | 메모리 RAM | 저장장치 SSD/HDD |
|---|---|---|
| 속도 | 빠름 | 느림 |
| 전원 종료 시 | 데이터 사라짐 | 데이터 유지 |
| 용도 | 실행 중 데이터 저장 | 파일 저장 |
| 예시 | 실행 중인 프로그램 | 사진, 코드 파일, DB 파일 |

프로그램은 SSD에 저장되어 있다가 실행될 때 RAM으로 올라온다.

---

# 프로그램 실행과 메모리 적재

프로그램 파일은 평소에는 저장장치에 있다.

실행하면 운영체제가 프로그램을 메모리에 올린다.

```text
SSD에 저장된 프로그램
↓
운영체제가 RAM으로 로드
↓
프로세스 생성
↓
CPU가 실행
```

즉, 실행 중인 프로그램은 메모리를 반드시 사용한다.

---

# 프로세스 메모리 구조

프로세스는 메모리를 여러 영역으로 나누어 사용한다.

대표적인 구조:

```text
높은 주소
┌──────────────┐
│ Stack        │
├──────────────┤
│ Heap         │
├──────────────┤
│ Data         │
├──────────────┤
│ Code         │
└──────────────┘
낮은 주소
```

각 영역은 역할이 다르다.

---

# Code 영역

Code 영역은 실행할 프로그램 명령어가 저장되는 공간이다.

예를 들어 C, Python, JavaScript 코드가 최종적으로 실행 가능한 형태로 변환되면 이 영역에 올라간다.

특징:
- 읽기 전용인 경우가 많음
- 프로그램 명령어 저장
- 실행 중 자주 바뀌지 않음

---

# Data 영역

Data 영역은 전역 변수와 static 변수가 저장되는 영역이다.

예시:

```c
int count = 10;
static int value = 20;
```

이런 변수들이 Data 영역에 저장된다.

특징:
- 프로그램 시작 시 할당
- 프로그램 종료 시 해제

---

# Stack 영역

Stack 영역은 함수 호출과 관련된 데이터를 저장한다.

저장되는 것:
- 지역 변수
- 함수 매개변수
- 반환 주소
- 이전 함수 상태

예시:

```python
def add(a, b):
    result = a + b
    return result
```

`a`, `b`, `result` 같은 함수 내부 값은 일반적으로 Stack 개념과 연결된다.

특징:
- 함수 호출 시 생성
- 함수 종료 시 자동 해제
- 속도가 빠름
- 크기가 제한적

---

# Heap 영역

Heap 영역은 동적으로 생성되는 데이터가 저장되는 공간이다.

예시:

```python
data = []
data.append(1)
data.append(2)
```

리스트, 객체, 동적 데이터는 Heap과 관련이 깊다.

특징:
- 실행 중 필요할 때 할당
- 직접 또는 GC가 해제
- 크기가 상대적으로 큼
- 관리가 잘못되면 Memory Leak 발생 가능

---

# Stack과 Heap 차이

| 항목 | Stack | Heap |
|---|---|---|
| 용도 | 함수 호출, 지역 변수 | 동적 데이터, 객체 |
| 관리 | 자동 | 수동 또는 GC |
| 속도 | 빠름 | 상대적으로 느림 |
| 크기 | 작음 | 큼 |
| 문제 | Stack Overflow | Memory Leak |

---

# Stack Overflow

Stack Overflow는 Stack 영역이 가득 차는 현상이다.

대표 원인:
- 재귀 함수가 너무 깊게 호출됨

예시:

```python
def func():
    func()

func()
```

이 코드는 자기 자신을 계속 호출하기 때문에 Stack이 계속 쌓이고, 결국 Stack Overflow가 발생한다.

---

# Memory Leak

Memory Leak은 더 이상 필요 없는 메모리를 해제하지 못하고 계속 잡고 있는 현상이다.

예시:

```python
data = []

while True:
    data.append("large data")
```

이 코드는 리스트에 데이터를 계속 추가한다.

삭제하지 않으면 메모리 사용량이 계속 증가한다.

이번 미션의 OOM Crash는 이 Memory Leak과 직접 연결된다.

---

# OOM

OOM은 Out Of Memory의 약자이다.

메모리가 부족한 상태를 의미한다.

OOM이 발생하면:
- 프로그램이 비정상 종료될 수 있음
- 운영체제가 프로세스를 강제 종료할 수 있음
- 시스템 전체가 느려질 수 있음

Linux에서는 OOM Killer가 동작할 수 있다.

이번 미션에서는 애플리케이션 내부 MemoryGuard가 메모리 초과를 감지하고 종료하는 흐름을 관찰한다.

---

# 가상 메모리

가상 메모리는 프로세스가 자신만의 독립된 메모리 공간을 가진 것처럼 보이게 하는 기술이다.

실제로는 물리 메모리 RAM을 여러 프로세스가 나눠 쓰지만, 각 프로세스는 자기만의 주소 공간을 가진 것처럼 동작한다.

장점:
- 프로세스 간 메모리 보호
- 큰 메모리를 쓰는 것처럼 보이게 함
- 메모리 관리 효율 증가

---

# 가상 주소와 물리 주소

프로그램이 사용하는 주소는 보통 가상 주소이다.

실제 RAM 주소는 물리 주소이다.

운영체제와 MMU가 가상 주소를 물리 주소로 변환한다.

```text
프로그램의 가상 주소
↓
MMU 주소 변환
↓
실제 RAM 물리 주소
```

이 구조 덕분에 한 프로세스가 다른 프로세스의 메모리에 함부로 접근하지 못한다.

---

# MMU

MMU는 Memory Management Unit이다.

CPU 안에 있는 메모리 주소 변환 장치이다.

역할:
- 가상 주소를 물리 주소로 변환
- 메모리 접근 권한 확인
- 프로세스 간 메모리 보호 지원

---

# 페이지

운영체제는 메모리를 페이지라는 일정한 크기의 블록으로 관리한다.

보통 4KB 단위가 많이 사용된다.

프로세스의 가상 메모리도 페이지 단위로 나뉘고, 물리 메모리도 페이지 프레임 단위로 나뉜다.

---

# 페이지 테이블

페이지 테이블은 가상 주소와 물리 주소의 매핑 정보를 저장하는 자료구조이다.

```text
가상 페이지 번호 → 물리 페이지 번호
```

운영체제는 각 프로세스마다 페이지 테이블을 관리한다.

---

# Page Fault

Page Fault는 프로세스가 접근하려는 메모리 페이지가 현재 RAM에 없을 때 발생한다.

이때 운영체제는:
- 디스크에서 필요한 페이지를 가져오거나
- 잘못된 접근이면 프로그램을 종료한다.

Page Fault가 너무 많이 발생하면 성능이 크게 떨어진다.

---

# Swap

Swap은 RAM이 부족할 때 디스크 일부를 메모리처럼 사용하는 기능이다.

장점:
- 메모리 부족 상황을 어느 정도 버틸 수 있음

단점:
- 디스크는 RAM보다 훨씬 느림
- Swap이 많이 발생하면 시스템이 심하게 느려짐

Memory Leak이 심하면 Swap 사용량이 늘고, 결국 시스템 전체가 느려질 수 있다.

---

# 캐시 메모리

캐시 메모리는 CPU와 RAM 사이에 있는 매우 빠른 메모리이다.

CPU는 RAM보다 훨씬 빠르다.

그래서 CPU가 매번 RAM에 접근하면 느리다.

이를 줄이기 위해 CPU 근처에 캐시를 둔다.

캐시 종류:
- L1 Cache
- L2 Cache
- L3 Cache

속도:
```text
레지스터 > L1 > L2 > L3 > RAM > SSD
```

---

# 지역성 원리

캐시는 지역성 원리를 이용한다.

## 시간 지역성

최근 사용한 데이터는 곧 다시 사용할 가능성이 높다.

## 공간 지역성

어떤 데이터를 사용하면 그 주변 데이터도 곧 사용할 가능성이 높다.

예를 들어 배열을 순서대로 읽으면 캐시 효율이 좋다.

---

# 프로세스

프로세스는 실행 중인 프로그램이다.

예:
- Python 프로그램 실행
- Node.js 서버 실행
- Chrome 실행
- Docker 컨테이너 안의 메인 프로그램 실행

프로세스는 운영체제로부터 자원을 할당받는다.

할당받는 자원:
- PID
- 메모리 공간
- 파일 디스크립터
- CPU 시간
- 환경변수
- 권한

---

# PID

PID는 Process ID이다.

운영체제가 프로세스에 부여하는 고유 번호이다.

확인:

```bash
ps -ef
```

특정 프로세스 검색:

```bash
ps -ef | grep agent
```

---

# 부모 프로세스와 자식 프로세스

프로세스는 다른 프로세스를 생성할 수 있다.

생성한 쪽:
- 부모 프로세스

생성된 쪽:
- 자식 프로세스

확인:

```bash
pstree
```

예시:

```text
bash
 └─ python
```

bash에서 python을 실행하면 bash가 부모, python이 자식 프로세스가 된다.

---

# 프로세스 상태

프로세스는 여러 상태를 가진다.

대표 상태:
- Running
- Ready
- Waiting
- Stopped
- Zombie

---

# Running

CPU에서 실제로 실행 중인 상태이다.

---

# Ready

실행 준비는 되었지만 CPU 차례를 기다리는 상태이다.

---

# Waiting

입출력, 락, 이벤트 등을 기다리는 상태이다.

Deadlock 상태의 스레드나 프로세스는 Waiting 상태와 관련이 깊다.

---

# Stopped

일시 중지된 상태이다.

---

# Zombie

프로세스는 종료되었지만 부모 프로세스가 종료 상태를 회수하지 않은 상태이다.

좀비 프로세스는 실제로 실행되지는 않지만 프로세스 테이블에 남아 있다.

---

# 스레드

스레드는 프로세스 내부의 실행 흐름이다.

하나의 프로세스 안에는 여러 스레드가 있을 수 있다.

```text
프로세스
├─ 스레드 A
├─ 스레드 B
└─ 스레드 C
```

스레드는 같은 프로세스의 메모리 공간을 공유한다.

공유:
- Heap
- Data 영역
- Code 영역
- 파일 디스크립터

각자 따로 가짐:
- Stack
- 레지스터 상태

---

# 멀티스레드

멀티스레드는 하나의 프로세스 안에서 여러 스레드를 사용하는 방식이다.

장점:
- 여러 작업을 동시에 처리 가능
- 응답성 향상
- 자원 공유 쉬움

단점:
- 동기화 문제
- Race Condition
- Deadlock
- 디버깅 어려움

---

# Race Condition

Race Condition은 여러 스레드가 같은 데이터를 동시에 접근하면서 실행 순서에 따라 결과가 달라지는 문제이다.

예시:

```python
count = 0

# Thread A
count += 1

# Thread B
count += 1
```

겉보기에는 결과가 2가 되어야 할 것 같지만, 실제로는 동시에 접근하면 1이 될 수도 있다.

이유는 `count += 1`이 한 번에 실행되는 명령이 아니기 때문이다.

실제로는 대략 다음 과정이다.

```text
count 읽기
1 더하기
count에 저장하기
```

두 스레드가 동시에 읽으면 결과가 꼬일 수 있다.

---

# Critical Section

Critical Section은 여러 스레드가 동시에 접근하면 문제가 생길 수 있는 코드 영역이다.

예:
- 공유 변수 수정
- 파일 쓰기
- DB 업데이트
- 계좌 잔액 변경

Critical Section은 보호해야 한다.

---

# Mutex

Mutex는 Mutual Exclusion의 줄임말이다.

한 번에 하나의 스레드만 Critical Section에 들어가게 하는 도구이다.

```text
스레드 A가 Lock 획득
↓
스레드 B는 대기
↓
스레드 A가 Lock 해제
↓
스레드 B가 Lock 획득
```

---

# Semaphore

Semaphore는 동시에 접근 가능한 스레드 수를 제한하는 동기화 도구이다.

Mutex가 1명만 들어가는 화장실이라면, Semaphore는 여러 명까지 들어갈 수 있는 공간이다.

예:
- DB 커넥션 풀 최대 10개
- 다운로드 동시 작업 최대 5개

---

# Deadlock

Deadlock은 여러 스레드나 프로세스가 서로 상대방이 가진 자원을 기다리며 영원히 멈추는 상태이다.

예시:

```text
Thread A: Lock 1 획득 후 Lock 2 기다림
Thread B: Lock 2 획득 후 Lock 1 기다림
```

결과:
- A는 B가 Lock 2를 놓기를 기다림
- B는 A가 Lock 1을 놓기를 기다림
- 둘 다 영원히 대기

---

# Deadlock 4대 조건

Deadlock이 발생하려면 보통 다음 네 조건이 동시에 만족된다.

## Mutual Exclusion

자원을 한 번에 하나만 사용할 수 있다.

## Hold and Wait

자원을 가진 상태에서 다른 자원을 기다린다.

## No Preemption

운영체제가 강제로 자원을 빼앗을 수 없다.

## Circular Wait

자원을 기다리는 관계가 원형 구조를 이룬다.

```text
A는 B를 기다림
B는 C를 기다림
C는 A를 기다림
```

---

# Deadlock 해결 방법

Deadlock 해결 방법은 크게 다음과 같다.

- 예방
- 회피
- 탐지 후 복구
- 무시

---

# Deadlock 예방

Deadlock 4대 조건 중 하나를 깨는 방식이다.

예:
- Lock 획득 순서 고정
- 자원을 한 번에 모두 요청
- Timeout 설정
- Lock 사용 최소화

---

# Deadlock 회피

시스템이 안전한 상태인지 확인하고 자원을 할당한다.

대표적으로 은행원 알고리즘이 있다.

---

# Deadlock 탐지 후 복구

Deadlock이 발생했는지 탐지한 뒤, 특정 프로세스를 종료하거나 자원을 회수한다.

---

# Deadlock 무시

일반 운영체제에서는 Deadlock이 자주 발생하지 않는다고 보고 무시하는 전략도 사용한다.

하지만 서버 프로그램에서는 Deadlock이 발생하면 서비스 장애로 이어질 수 있다.

---

# 이번 미션에서 Deadlock을 관찰하는 방법

Deadlock은 프로그램이 종료되지 않는다는 점이 특징이다.

확인 포인트:
- PID가 살아 있음
- CPU 사용량 변화 없음
- 메모리 변화 없음
- 로그가 더 이상 출력되지 않음
- 마지막 로그에 WAITING, BLOCKED 같은 단서 존재

관련 명령어:

```bash
ps -ef | grep agent
```

```bash
top -H
```

```bash
ps -L -p PID
```

```bash
tail -f app.log
```

---

# 운영체제

운영체제는 컴퓨터 하드웨어와 응용 프로그램 사이에서 자원을 관리하는 시스템 소프트웨어이다.

대표 운영체제:
- Linux
- Windows
- macOS
- Android
- iOS

운영체제의 주요 역할:
- 프로세스 관리
- 메모리 관리
- 파일 시스템 관리
- 입출력 관리
- 네트워크 관리
- 보안과 권한 관리

---

# 커널

커널은 운영체제의 핵심 부분이다.

커널은 하드웨어 자원을 직접 관리한다.

역할:
- CPU 스케줄링
- 메모리 관리
- 프로세스 관리
- 파일 시스템 관리
- 장치 드라이버 관리

사용자 프로그램은 커널에 직접 접근하지 않고 시스템 콜을 통해 요청한다.

---

# 사용자 모드와 커널 모드

CPU는 보통 두 가지 권한 모드로 동작한다.

- 사용자 모드
- 커널 모드

## 사용자 모드

일반 프로그램이 실행되는 모드이다.

제한이 많다.

프로그램이 하드웨어를 직접 제어할 수 없다.

## 커널 모드

운영체제 커널이 실행되는 모드이다.

하드웨어와 메모리에 직접 접근할 수 있다.

---

# 시스템 콜

시스템 콜은 사용자 프로그램이 운영체제 커널에게 서비스를 요청하는 방법이다.

예:
- 파일 열기
- 파일 읽기
- 프로세스 생성
- 메모리 할당
- 네트워크 송신

예시 흐름:

```text
프로그램이 파일 읽기 요청
↓
read() 시스템 콜 호출
↓
커널 모드 진입
↓
커널이 디스크에서 데이터 읽음
↓
사용자 프로그램으로 결과 반환
```

---

# 왜 시스템 콜이 필요한가

프로그램이 하드웨어를 직접 제어하면 위험하다.

예:
- 다른 프로세스 메모리 침범
- 디스크 데이터 손상
- 네트워크 장치 오작동

그래서 운영체제가 중간에서 통제한다.

---

# 파일 디스크립터

파일 디스크립터는 프로세스가 열고 있는 파일이나 입출력 자원을 가리키는 번호이다.

Linux에서는 거의 모든 것을 파일처럼 다룬다.

예:
- 일반 파일
- 터미널
- 소켓
- 파이프

기본 파일 디스크립터:

| 번호 | 의미 |
|---|---|
| 0 | stdin |
| 1 | stdout |
| 2 | stderr |

---

# 표준 입출력

## stdin

표준 입력이다.

보통 키보드 입력이다.

## stdout

표준 출력이다.

보통 터미널 출력이다.

## stderr

표준 에러 출력이다.

에러 메시지 출력에 사용된다.

---

# 리다이렉션

출력을 파일로 저장할 수 있다.

```bash
command > output.txt
```

기존 내용을 덮어쓴다.

```bash
command >> output.txt
```

기존 내용 뒤에 추가한다.

에러 출력 저장:

```bash
command 2> error.log
```

---

# 파이프

파이프는 한 명령어의 출력을 다른 명령어의 입력으로 넘기는 기능이다.

예:

```bash
ps -ef | grep agent
```

의미:
- `ps -ef` 결과를
- `grep agent`의 입력으로 전달

---

# CPU 스케줄링

CPU 스케줄링은 운영체제가 어떤 프로세스나 스레드에게 CPU를 줄지 결정하는 과정이다.

CPU는 한정된 자원이다.

프로세스는 많고 CPU 코어는 제한적이다.

그래서 운영체제는 공정하고 효율적으로 CPU를 나누어야 한다.

---

# 스케줄러

스케줄러는 다음에 실행할 프로세스나 스레드를 선택하는 운영체제 구성 요소이다.

고려 요소:
- 우선순위
- 대기 시간
- CPU 사용 시간
- 응답성
- 공정성

---

# Context Switching

Context Switching은 CPU가 실행 중인 작업을 바꾸는 과정이다.

예:

```text
Thread A 실행
↓
Thread A 상태 저장
↓
Thread B 상태 복원
↓
Thread B 실행
```

저장되는 정보:
- 레지스터 값
- Program Counter
- Stack Pointer
- 프로세스 상태

Context Switching은 필요하지만 비용이 든다.

너무 자주 발생하면 오히려 성능이 떨어진다.

---

# Time Slice

Time Slice는 각 프로세스나 스레드가 CPU를 사용할 수 있는 짧은 시간 단위이다.

Round Robin 스케줄링에서 중요하다.

예:
- Thread A 10ms 실행
- Thread B 10ms 실행
- Thread C 10ms 실행

이렇게 빠르게 번갈아 실행하면 동시에 실행되는 것처럼 보인다.

---

# FCFS

FCFS는 First Come First Served이다.

먼저 온 작업을 먼저 처리한다.

장점:
- 단순함
- 구현 쉬움

단점:
- 긴 작업이 앞에 있으면 뒤 작업들이 오래 기다림

이를 Convoy Effect라고 한다.

---

# Round Robin

Round Robin은 각 프로세스에게 일정 시간씩 CPU를 나누어 주는 방식이다.

장점:
- 공정함
- 응답성이 좋음
- 시분할 시스템에 적합

단점:
- Time Slice가 너무 짧으면 Context Switching 비용 증가
- Time Slice가 너무 길면 FCFS와 비슷해짐

이번 미션의 보너스 과제에서 로그 패턴을 보고 Round Robin인지 추론할 수 있다.

---

# Priority Scheduling

Priority Scheduling은 우선순위가 높은 작업을 먼저 실행하는 방식이다.

장점:
- 중요한 작업을 빠르게 처리 가능

단점:
- 낮은 우선순위 작업이 계속 밀릴 수 있음

이를 Starvation이라고 한다.

---

# Starvation

Starvation은 특정 프로세스가 자원을 오랫동안 받지 못하는 상태이다.

예:
- 우선순위가 낮은 작업이 계속 CPU를 받지 못함

해결 방법:
- Aging

---

# Aging

Aging은 오래 기다린 프로세스의 우선순위를 점점 높여주는 방식이다.

Starvation을 완화하기 위해 사용된다.

---

# I/O Bound와 CPU Bound

프로그램은 성격에 따라 크게 나눌 수 있다.

## CPU Bound

CPU 계산이 많은 작업이다.

예:
- 영상 인코딩
- AI 모델 추론
- 수학 계산
- 암호화

## I/O Bound

입출력 대기가 많은 작업이다.

예:
- 파일 읽기
- DB 요청
- 네트워크 요청
- 사용자 입력 대기

CPU Bound 작업은 CPU 사용률이 높아질 수 있다.

I/O Bound 작업은 Waiting 상태가 많을 수 있다.

---

# Busy Waiting

Busy Waiting은 실제로 할 일은 없지만 조건이 만족될 때까지 CPU를 계속 사용하며 기다리는 방식이다.

예:

```python
while not ready:
    pass
```

이 코드는 기다리기만 하는데 CPU를 계속 사용한다.

CPU Spike의 원인이 될 수 있다.

---

# Blocking

Blocking은 어떤 작업이 끝날 때까지 대기 상태로 들어가는 것이다.

예:
- 파일 읽기 대기
- 네트워크 응답 대기
- Lock 대기

Blocking 상태에서는 보통 CPU를 많이 사용하지 않는다.

Deadlock 상태도 Blocking과 관련이 있다.

---

# Interrupt

Interrupt는 CPU가 현재 작업을 잠시 멈추고 중요한 이벤트를 처리하게 만드는 신호이다.

예:
- 키보드 입력
- 네트워크 패킷 도착
- 디스크 작업 완료
- 타이머 이벤트

운영체제는 Interrupt를 통해 하드웨어 이벤트를 처리한다.

---

# Timer Interrupt

Timer Interrupt는 일정 시간마다 발생하는 인터럽트이다.

운영체제는 Timer Interrupt를 이용해서 CPU를 한 프로세스가 독점하지 못하게 한다.

예:
- A 프로세스 실행
- Timer Interrupt 발생
- 운영체제가 제어권 회수
- B 프로세스 실행

이것이 선점형 스케줄링의 기반이다.

---

# 선점형 스케줄링

선점형 스케줄링은 운영체제가 실행 중인 프로세스의 CPU를 강제로 회수할 수 있는 방식이다.

현대 운영체제는 대부분 선점형 스케줄링을 사용한다.

장점:
- 한 프로세스가 CPU를 독점하지 못함
- 응답성 향상

---

# 비선점형 스케줄링

비선점형 스케줄링은 실행 중인 프로세스가 스스로 CPU를 반납할 때까지 기다리는 방식이다.

장점:
- Context Switching 적음

단점:
- 하나의 작업이 CPU를 오래 점유할 수 있음

# MEMORY_LIMIT

`MEMORY_LIMIT`은 메모리 사용 제한값이다.

예:

```bash

```

의미:
- 프로세스가 256MB 근처까지 메모리를 사용하면 MemoryGuard가 종료할 수 있음

값을 높이면:
- 더 오래 실행될 수 있음
- 하지만 Memory Leak 자체가 해결되는 것은 아님

---

# CPU_MAX_OCCUPY

`CPU_MAX_OCCUPY`는 CPU 사용률 제한값이다.

예:

```bash
export CPU_MAX_OCCUPY=80
```

의미:
- CPU 사용률이 80%를 초과하면 Watchdog이 보호 조치를 할 수 있음

값을 높이면:
- 종료 시점이 늦어질 수 있음
- 하지만 CPU 폭주 원인이 해결되는 것은 아님

---

# MULTI_THREAD_ENABLE

`MULTI_THREAD_ENABLE`은 멀티스레드 실행 여부를 결정하는 값이다.

예:

```bash
export MULTI_THREAD_ENABLE=true
```

또는:

```bash
export MULTI_THREAD_ENABLE=false
```

true일 때:
- 여러 스레드 실행
- Deadlock 재현 가능성 있음

false일 때:
- 단일 스레드 실행
- Deadlock 회피 가능성 있음

# Before & After 비교

Before & After는 조치 전과 조치 후를 비교하는 것이다.

예:

```text
Before:
MEMORY_LIMIT=256
10분 후 종료

After:
MEMORY_LIMIT=512
30분 이상 생존
```

중요한 점:
- 조치 후 좋아졌는지 수치로 보여야 함
- 단순히 “괜찮아진 것 같다”는 부족함

---

# Workaround

Workaround는 임시 조치이다.

예:
- MEMORY_LIMIT 증가
- CPU_MAX_OCCUPY 증가
- MULTI_THREAD_ENABLE=false 설정

중요:
Workaround는 근본 해결이 아닐 수 있다.

예:
- 메모리 제한을 늘려도 Memory Leak 자체는 남아 있음
- 멀티스레드를 끄면 Deadlock은 피하지만 성능이 낮아질 수 있음

---

# Root Cause

Root Cause는 근본 원인이다.

예:

나쁜 분석:
```text
프로그램이 죽었다.
```

좋은 분석:
```text
Heap 영역에 데이터가 지속적으로 누적되면서 메모리 사용량이 증가했고,
MEMORY_LIMIT에 도달하자 MemoryGuard가 프로세스를 종료했다.
```

Root Cause Analysis는 현상보다 한 단계 더 깊게 들어가야 한다.

# 장애 분석 흐름

```text

↓
관제 데이터 수집
↓
로그 확인
↓
원인 후보 세우기
↓

↓
재실행
↓
Before & After 비교
↓
리포트 작성
```

# OOM Crash 분석 흐름

OOM 분석 시 봐야 할 것:

```text
메모리 사용량이 시간에 따라 증가하는가?
↓
종료 직전 메모리 수치는 얼마인가?
↓
로그에 Memory limit exceeded가 있는가?
↓
MEMORY_LIMIT 값을 바꾸면 생존 시간이 바뀌는가?
```

핵심:
- 메모리 증가 패턴
- 종료 로그
- MEMORY_LIMIT 전후 비교

---

# CPU Latency 분석 흐름

CPU 분석 시 봐야 할 것:

```text
CPU 사용률이 갑자기 증가하는가?
↓
시스템 전체 문제인가, 특정 프로세스 문제인가?
↓
top/ps에서 agent 프로세스 CPU가 높은가?
↓
로그에 Watchdog 메시지가 있는가?
↓
CPU_MAX_OCCUPY 값을 바꾸면 종료 시점이 바뀌는가?
```

핵심:
- 특정 프로세스 CPU 과점유
- Watchdog 보호 조치
- CPU_MAX_OCCUPY 전후 비교

---

# Deadlock 분석 흐름

Deadlock 분석 시 봐야 할 것:

```text
프로세스 PID가 살아 있는가?
↓
CPU 사용량이 거의 없는가?
↓
메모리 변화가 없는가?
↓
로그가 멈췄는가?
↓
마지막 로그에 WAITING/BLOCKED 단서가 있는가?
↓
MULTI_THREAD_ENABLE=false로 바꾸면 회피되는가?
```

핵심:
- 종료가 아니라 무응답
- PID 존재
- 로그 정지
- 스레드/락 대기 추론

# MemoryGuard

MemoryGuard는 애플리케이션 내부에서 메모리 사용량을 감시하는 보호 장치라고 볼 수 있다.

메모리 사용량이 제한값을 초과하면 프로세스를 종료해서 시스템 전체 장애를 막는다.

중요:
MemoryGuard가 종료했다는 것은 메모리 문제가 없다는 뜻이 아니다.

오히려:
- 메모리 누수가 있어서 보호 정책이 개입했다는 뜻

이다.

# Watchdog

Watchdog은 프로그램이나 시스템이 비정상 상태에 빠졌는지 감시하는 장치이다.

조건:
- CPU 과점유
- 무응답
- 제한 시간 초과
- 상태 체크 실패

문제가 감지되면:
- 경고 로그 출력
- 프로세스 종료
- 재시작
- 알림 전송

등을 수행할 수 있다.

이번 미션에서는 CPU 과점유 보호 조치와 연결된다.

---

# Latency

Latency는 지연 시간을 의미한다.

CPU Latency는 CPU 과점유로 인해 작업 처리가 늦어지는 상황과 관련된다.

예:
- 요청 응답이 늦어짐
- 스레드가 CPU를 못 받음
- 다른 프로세스 실행이 밀림

---

# Throughput

Throughput은 처리량이다.

예:
- 초당 요청 처리 수
- 초당 작업 완료 수

Latency와 Throughput은 둘 다 성능 분석에서 중요하다.

---

# Latency와 Throughput 차이

| 항목 | 의미 | 예시 |
|---|---|---|
| Latency | 하나의 작업이 완료되기까지 걸리는 시간 | 요청 하나가 2초 걸림 |
| Throughput | 단위 시간당 처리량 | 초당 100개 요청 처리 |

CPU Spike가 발생하면 Latency가 증가할 수 있다.

---

# Load Average

Load Average는 실행 중이거나 실행 대기 중인 프로세스 수의 평균이다.

`top` 명령어에서 볼 수 있다.

예:

```text
load average: 1.50, 2.00, 2.30
```

순서:
- 1분 평균
- 5분 평균
- 15분 평균

CPU 코어 수와 함께 봐야 한다.

예:
- 1코어에서 Load Average 4는 높음
- 8코어에서 Load Average 4는 상대적으로 괜찮을 수 있음

---

# RSS와 VSZ

Linux에서 프로세스 메모리를 볼 때 RSS와 VSZ가 자주 나온다.

## VSZ

Virtual Memory Size

프로세스가 사용할 수 있다고 예약한 가상 메모리 크기이다.

## RSS

Resident Set Size

실제로 RAM에 올라와 있는 메모리 크기이다.

미션에서 실제 메모리 점유를 볼 때는 RSS가 더 중요할 수 있다.

---

# 버퍼와 캐시

Linux에서 `free -h`를 보면 buff/cache가 나온다.

Linux는 남는 메모리를 디스크 캐시로 활용한다.

그래서 단순히 “used가 높다”고 해서 무조건 문제는 아니다.

중요한 것은:
- available 메모리
- 특정 프로세스의 RSS 증가
- Swap 사용 증가

이다.

# Docker와 운영체제

Docker 컨테이너는 가상머신이 아니라, 호스트 운영체제의 커널을 공유하는 격리된 프로세스 환경이다.

즉 컨테이너 안의 프로그램도 결국 호스트 입장에서는 프로세스이다.

Docker에서 중요한 OS 개념:
- 프로세스
- 네임스페이스
- cgroups
- 파일 시스템 격리
- 네트워크 격리

---

# cgroups

cgroups는 Linux에서 프로세스의 자원 사용량을 제한하고 관리하는 기능이다.

제한 가능:
- CPU
- 메모리
- 디스크 I/O
- 네트워크

Docker의 `--memory`, `--cpus` 같은 옵션이 cgroups와 관련된다.

---

# Namespace

Namespace는 프로세스가 보는 시스템 자원을 격리하는 기능이다.

예:
- PID Namespace
- Network Namespace
- Mount Namespace
- User Namespace

컨테이너가 독립된 환경처럼 보이는 이유가 Namespace 때문이다.

# 컴퓨터구조 관점에서 미션 이해

## OOM Crash

컴퓨터구조 관점:
- 프로세스가 Heap 메모리를 계속 사용
- RAM 사용량 증가
- 제한값 도달
- 보호 정책 개입

## CPU Spike

컴퓨터구조 관점:
- CPU 코어가 특정 연산에 과도하게 사용됨
- 스케줄러가 다른 작업에 CPU를 배분하기 어려워짐
- Latency 증가

## Deadlock

컴퓨터구조/운영체제 관점:
- 여러 스레드가 공유 자원을 기다림
- CPU는 놀고 있지만 프로그램은 진행 불가
- 프로세스는 살아 있으나 작업이 멈춤

---

# 운영체제 관점에서 미션 이해

## OOM Crash

운영체제는 프로세스별 메모리 사용량을 관리한다.

메모리 사용량이 비정상적으로 증가하면:
- 시스템 전체 성능 저하
- Swap 증가
- OOM 위험

이 발생한다.

## CPU Latency

운영체제는 여러 프로세스에게 CPU 시간을 배분한다.

특정 프로세스가 CPU를 과도하게 사용하면:
- 다른 프로세스 응답 지연
- 시스템 부하 증가
- Watchdog 동작

이 발생할 수 있다.

## Deadlock

운영체제는 스레드 상태를 관리하지만, 애플리케이션 내부 Lock 설계 문제까지 자동으로 해결해주지는 못한다.

따라서 개발자가 로그와 상태를 보고 Deadlock을 추론해야 한다.

# 미션 리포트를 잘 쓰기 위한 관점

좋은 리포트는 다음 구조를 가진다.

```text
현상
↓
증거
↓
원인
↓
조치
↓
검증
```

나쁜 리포트:

```text
프로그램이 꺼졌다.
MEMORY_LIMIT을 올렸다.
해결됐다.
```

좋은 리포트:

```text
monitor.sh 관제 결과 메모리 사용량이 시간에 따라 지속적으로 증가했다.
종료 직전 로그에서 Memory limit exceeded 메시지가 확인되었다.
따라서 Heap 영역에 데이터가 누적되는 Memory Leak으로 추정된다.
MEMORY_LIMIT을 256MB에서 512MB로 조정한 결과 생존 시간이 증가했다.
하지만 근본 해결은 누수 지점의 메모리 해제 또는 데이터 누적 구조 개선이다.
```

- MULTI_THREAD_ENABLE 변경 전후 비교했는가?

# 이론을 공부할 때 가장 중요한 연결

이 미션에서 이론은 따로 외우는 것이 아니라 아래처럼 연결해서 이해해야 한다.

```text
프로세스
→ PID로 관찰 가능
→ ps/top으로 확인

메모리
→ Heap 증가
→ Memory Leak
→ OOM Crash

CPU
→ 특정 프로세스 과점유
→ CPU Spike
→ Watchdog 종료

스레드
→ 공유 자원 접근
→ Lock
→ Deadlock

스케줄링
→ CPU 분배
→ Context Switching
→ Round Robin 추론

환경변수
→ 실행 설정 변경
→ Before & After 검증
```

# 이번 미션에서 가장 중요한 한 문장

```text
현상만 보는 사람 → 초보자
증거를 모으는 사람 → 실무자
원인을 설명하고 재발 방지까지 제안하는 사람 → 좋은 개발자
```
